# Version 1.0 Master Checklist

## 1. Purpose

This is the master project tracker for everything between the current Beta-1 state and a real Version 1.0 release of Jarvis, Nexus, and CommandCore. It follows `docs/vision/Jarvis-Nexus-Experience-Vision.md` for what "done" ultimately means, and consolidates status across every roadmap, backlog, and architecture document produced so far. This document should be updated as a living tracker, not re-derived from scratch each time — when a stream completes work, update this file's row for the relevant category rather than creating a parallel status document.

## 2. How To Read This Table

- **Current Status** — a short, honest description of what exists today, not what is planned.
- **Dependencies** — what this category needs to land before it can progress further.
- **Completion %** — a rough estimate scoped to "progress toward a real Version 1.0," not toward any single wave. A category can be 100% done for Beta-1's scope while still being 10% of its full Version 1.0 scope — both numbers matter, so both are noted where they differ.
- **Target Milestone** — Beta-1, Beta-2, or V1 (or "Unscoped" where no milestone has claimed it yet).
- **Owner Stream** — which sprint/stream produced the current state or documentation, for traceability back to source material.

## 4. Notes On Unscoped Items

- **Glassmind** and **NightShift** appear in this tracker because they were named as Version 1.0 tracking categories, but no prior document in this codebase defines what they are. They are listed here as placeholders with explicit "Unscoped" status rather than given fabricated descriptions. Before either can move off 0%, a defining document analogous to the Planetary Operating Model or the Jarvis Conversation Architecture needs to be written and reviewed.
- **Multi-user** is similarly unscoped beyond the single-operator assumption that runs through every Beta-1/Beta-2 document to date. This should be explicitly addressed alongside Authentication and Permissions during Beta-2 scoping, not discovered as a gap after those two are already built.

## 5. Maintenance Of This Document

- Update the relevant row's **Current Status**, **Completion %**, and **Owner Stream** columns whenever a stream completes work touching that category — do not wait for a full re-audit.
- If a category's scope is split across multiple milestones (as several already are — "Beta-1 done; Beta-2 for X"), keep both halves of that status visible rather than collapsing to a single number that hides which part is actually finished.
- When **Glassmind**, **NightShift**, or **Multi-user** get their first defining document, update their rows immediately rather than leaving stale "Unscoped" status after the gap has been closed.
